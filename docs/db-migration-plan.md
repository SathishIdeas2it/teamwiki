# Database Migration Plan

## Overview

TeamWiki uses **Prisma Migrate** for all schema migrations. Migration files are generated by
`prisma migrate dev` and are **never edited by hand**, with one explicit exception: the initial
migration must have the full-text search trigger appended manually (documented in
[Step 3](#step-3-append-the-fts-trigger) below).

---

## Schema Models

| Model | Table | Soft Delete | Notes |
|---|---|---|---|
| `User` | `users` | `deleted_at TIMESTAMPTZ` | Preserves FK references from articles/revisions |
| `Account` | `accounts` | — | NextAuth provider accounts; hard-delete via Cascade |
| `Session` | `sessions` | — | NextAuth sessions; hard-delete via Cascade |
| `VerificationToken` | `verification_tokens` | — | Short-lived; hard-delete |
| `Category` | `categories` | — | Low-churn; hard-delete acceptable |
| `Tag` | `tags` | — | `category_id` SET NULL on category delete |
| `Article` | `articles` | `deleted_at TIMESTAMPTZ` | FTS column `search_vector` trigger-managed |
| `ArticleTag` | `article_tags` | — | Composite PK; CASCADE on article or tag delete |
| `ArticleRevision` | `article_revisions` | — | Immutable snapshots; CASCADE on article hard-delete |
| `AuditLog` | `audit_logs` | — | Append-only; `actor_id` SET NULL on user delete |

### Soft Delete Behaviour

Soft delete is applied only to `users` and `articles` — the two entities whose deletion would
cascade destructively to other data. The application layer filters these records with
`WHERE deleted_at IS NULL`.

```sql
-- Active users
SELECT * FROM users WHERE deleted_at IS NULL;

-- Active articles
SELECT * FROM articles WHERE deleted_at IS NULL AND status = 'PUBLISHED';
```

Restoring a soft-deleted record:
```sql
UPDATE users   SET deleted_at = NULL WHERE id = $1;
UPDATE articles SET deleted_at = NULL WHERE id = $1;
```

## Migration Steps

### Step 1 — Prerequisites

```bash
# Ensure DATABASE_URL points to your development database
cp .env.example .env
# Edit .env and set DATABASE_URL and DATABASE_URL_TEST
```

### Step 2 — Generate the Initial Migration

Run this command. Prisma inspects `prisma/schema.prisma`, diffs it against the (empty) database,
and writes the migration SQL into `prisma/migrations/<timestamp>_init/migration.sql`.

```bash
npx prisma migrate dev --name init
```

Prisma will:
1. Create all tables with the correct column types and constraints.
2. Create all foreign key relationships.
3. Create all `@@index` entries as B-tree indexes.
4. Apply the migration to the development database.

### Step 3 — Append the FTS Trigger

**This is the only permitted manual edit to a migration file.**

Open `prisma/migrations/<timestamp>_init/migration.sql` and append the following SQL **at the
end** of the file, after all `CREATE TABLE` and `CREATE INDEX` statements:

```sql
-- Full-text search: trigger-managed tsvector column on articles.
-- Weight A = title (highest rank), Weight B = content.
-- This trigger fires on INSERT and on UPDATE of title or content.

CREATE OR REPLACE FUNCTION articles_search_vector_update()
RETURNS trigger AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('english', coalesce(NEW.title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(NEW.content, '')), 'B');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER articles_search_vector_update
  BEFORE INSERT OR UPDATE OF title, content ON articles
  FOR EACH ROW EXECUTE FUNCTION articles_search_vector_update();
```

After appending, mark the migration as already applied (since `prisma migrate dev` already ran it
without the trigger):

```bash
# Resolve: apply just the trigger on the existing dev DB
npx prisma db execute --file prisma/migrations/<timestamp>_init/migration.sql
```

Or more precisely, extract just the trigger SQL into a standalone file and run it:

```bash
# Apply only the trigger additions
psql $DATABASE_URL -c "
CREATE OR REPLACE FUNCTION articles_search_vector_update()
RETURNS trigger AS \$\$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('english', coalesce(NEW.title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(NEW.content, '')), 'B');
  RETURN NEW;
END;
\$\$ LANGUAGE plpgsql;

CREATE TRIGGER articles_search_vector_update
  BEFORE INSERT OR UPDATE OF title, content ON articles
  FOR EACH ROW EXECUTE FUNCTION articles_search_vector_update();
"
```

### Step 4 — Populate the Search Vector for Existing Data

After the trigger is installed, any new INSERT/UPDATE automatically populates `search_vector`.
Existing rows need a one-time backfill:

```sql
UPDATE articles SET search_vector =
  setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
  setweight(to_tsvector('english', coalesce(content, '')), 'B');
```

### Step 5 — Seed Development Data

```bash
npx prisma db seed
```

This creates:
- SYSTEM service account (`system@teamwiki.internal`)
- Admin, Editor, and Viewer sample users
- Engineering and Process categories
- Backend, Frontend, and Onboarding tags
- Three sample articles with initial revision snapshots

### Step 6 — Set Up the Test Database

```bash
DATABASE_URL=$DATABASE_URL_TEST npx prisma migrate deploy
# Then apply the FTS trigger on the test DB (same SQL as Step 3)
psql $DATABASE_URL_TEST -c "<trigger SQL from Step 3>"
```

Integration tests create and destroy data using `resetDatabase()` in
`tests/integration/helpers/db.ts`. The SYSTEM user is created once in `globalSetup.ts`.

## Ongoing Migrations

For every subsequent schema change:

```bash
npx prisma migrate dev --name <descriptive-name>
```

Examples:
```bash
npx prisma migrate dev --name add-article-view-count
npx prisma migrate dev --name make-tag-description-required
```

### Destructive Change Policy

Follow the three-PR process to avoid downtime:

| PR | Migration | App code |
|---|---|---|
| 1 | Make column nullable / add new column | Read from both old and new |
| 2 | Backfill data | Switch writes to new column |
| 3 | Drop old column | Remove old-column references |

**Never** use `DROP COLUMN` in the same migration that adds the replacement column.

### Deploying to Production

Production migrations run via `prisma migrate deploy` (never `migrate dev`). This is triggered
in the GitHub Actions workflow after manual approval:

```bash
npx prisma migrate deploy
```

## Index Strategy

### Indexes Defined in Schema

| Table | Index | Type | Rationale |
|---|---|---|---|
| `articles` | `search_vector` | GIN (manual) | Required for `@@` FTS operator — B-tree cannot serve this |
| `articles` | `author_id` | B-tree | Filter articles by author |
| `articles` | `status` | B-tree | Filter PUBLISHED vs DRAFT vs ARCHIVED |
| `articles` | `published_at DESC` | B-tree | Sort articles list by publish date |
| `articles` | `deleted_at` | B-tree | Filter `WHERE deleted_at IS NULL` |
| `article_revisions` | `(article_id, revision_number DESC)` | B-tree | Fetch latest revision efficiently |
| `users` | `deleted_at` | B-tree | Filter `WHERE deleted_at IS NULL` |
| `tags` | `category_id` | B-tree | List tags by category |
| `accounts` | `user_id` | B-tree | NextAuth account lookup |
| `sessions` | `user_id` | B-tree | NextAuth session lookup |
| `audit_logs` | `actor_id` | B-tree | Audit trail by user |
| `audit_logs` | `created_at DESC` | B-tree | Paginate audit log |

### GIN Index (manual SQL — not expressible in Prisma schema)

Append this to `prisma/migrations/<timestamp>_init/migration.sql` alongside the trigger:

```sql
CREATE INDEX articles_search_vector_idx ON articles USING GIN(search_vector);
```

This index is required for the `WHERE search_vector @@ query` clause to use the fast GIN scan
rather than a sequential scan.

### Recommended Future Partial Index

Once soft-deletes accumulate, consider replacing the B-tree `deleted_at` index with a partial
index (requires raw SQL in a migration):

```sql
-- Drop the Prisma-generated B-tree index first
DROP INDEX IF EXISTS articles_deleted_at_idx;

-- Partial index that only indexes non-deleted rows (far smaller, faster for active queries)
CREATE INDEX articles_active_idx ON articles (created_at DESC) WHERE deleted_at IS NULL;
```

This is a future optimization — not required for launch.
